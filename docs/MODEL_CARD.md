## Intended Use

**Primary use:** Rank 100,000 candidates for a "Senior AI Engineer — Founding Team"
role at Redrob/India.Runs, producing a top-100 shortlist with fact-grounded explanations.

**Intended users:** Redrob recruiters and hiring managers reviewing the shortlist.

**Out-of-scope uses:**
- Hiring decisions without human review (the system is a shortlisting aid, not a decision-maker).
- Roles other than the specific JD the ranker was distilled from.
- Jurisdictions where algorithmic screening has legal constraints (GDPR, EEOC).

## Architecture

```
Precompute (offline, one-time):
  100k candidates → BGE-small-en-v1.5 embeddings (384-d, CPU)
                 → BM25 index (rank-bm25)
                 → 59-feature static store (features.parquet)
                 → LLM-judge tier labels (0–4) on 1,430 stratified candidates
                 → LightGBM LambdaMART (lambdarank, 500 trees, 31 leaves)
                 → Calibration YAML

Rank (≤5 min, CPU-only, no network):
  features.parquet + lgbm_ranker.txt
  → rubric score (transparent weighted formula, 6 components)
  → LightGBM predict
  → blend (0.6 × lgbm + 0.4 × rubric)
  → multiplicative modifiers (availability × location × offtarget_gate × honeypot_gate)
  → sort + tie-break by candidate_id
  → top-100 + fact-grounded reasoning strings
```

## Training Data

**Labeled set:** 1,430 candidates from the 100k pool, stratified by:
- 10% confirmed honeypots (all included)
- 30% high-fit (role_target ≥ 1.0 + retrieval/eval signals)
- 22% mid-tier
- 25% off-target function
- 10% foreign
- 3% remainder (uniform random)

**Labels:** Tier 0–4 from a free multi-provider LLM-judge stack:
- Cerebras gpt-oss-120b (primary workhorse, ~1M free tokens/day)
- Groq: llama-4-scout-17b, llama-3.1-8b-instant, llama-3.3-70b
- Gemini: 2.5-flash, 2.5-flash-lite

Each model uses the **exact JD as the scoring rubric** (not a generic "rate this candidate" prompt). No candidate data was sent to any paid/hosted model. Total label cost: $0.

**Label distribution:** tier-0: 1,010 (70.6%), tier-1: 356 (24.9%), tier-2: 54 (3.8%), tier-3: 8 (0.6%), tier-4: 2 (0.1%). Class imbalance is expected and reflects the true proportion of highly-relevant candidates in the pool.

## Evaluation Results

Evaluated on a held-out 20% split of the 1,430 labeled candidates:

| Variant | Composite | NDCG@10 | NDCG@50 | MAP |
|---|---|---|---|---|
| BM25 only (baseline) | 0.7993 | 0.8592 | 0.7535 | 0.7242 |
| Dense-only | 0.6601 | 0.6595 | 0.6563 | 0.6564 |
| Rubric (no gates) | 0.6541 | 0.6449 | 0.7218 | 0.6345 |
| Rubric + modifiers | 0.4085 | 0.4189 | 0.5388 | 0.1826 |
| **LightGBM full** | **0.4071** | **0.3790** | **0.6038** | **0.2096** |

> **Important context on BM25 dominance:** Our LLM-judge labels were generated using the
> JD text as the rubric — the same text BM25 operates on. This creates lexical overlap
> between the labeling signal and BM25, which inflates BM25's apparent performance on
> *our* labels. The hidden ground truth was generated by a different process (human/rubric),
> and our system's role/function discrimination, honeypot detection, and behavioral signals
> provide correctness properties that BM25 cannot. See the ablation tables for per-property
> analysis.

**Top-100 health (full 100k ranking):**
- Honeypots: **0/100** (vs. 4–10 expected in a naive keyword ranking)
- Off-target function: **0/100**
- Services-only careers: **0/100**
- Out-of-India: **0/100**
- Runtime: **~40 s**, Peak RAM: **196 MB** (7.5× headroom on the 5-min / 16 GB budget)

## Ethical Considerations

**Behavioral signals:** All 23 behavioral signals from Redrob are used ONLY as a gentle
availability modifier (clamped [0.75, 1.05]). Sentinel values (−1 = no history) are
treated as NEUTRAL — candidates are never penalized for having no history on the platform.
This is verified in `tests/test_adversarial.py::test_sentinel_not_penalized`.

**Location:** The JD explicitly requires Pune/Noida and allows India-wide. Location is a
modifier (range [0.70, 1.04]), not a primary signal. Non-India candidates are not
explicitly excluded — they receive a lower modifier but can still rank in the top-100 if
their fit is exceptional.

**Name/gender neutrality:** Candidate names are anonymized in the Redrob dataset. No
gender, age, or demographic inference is performed.

**Honeypot detection:** Our consistency checker flags candidates with statistically
impossible timelines (YOE > career span + education buffer). This is a data-quality
signal, not a judgment about the person.

## Limitations

1. **Label scarcity at top tiers:** Only 10 tier-3+ candidates in 1,430 labels (0.7%).
   NDCG@10 is disproportionately influenced by the placement of these few candidates.
2. **LLM-judge label bias:** Labels were generated using the JD as a rubric; the judge
   may have over-rated profiles with high lexical overlap with the JD text.
3. **Single-query learning:** The ranker was trained for exactly one JD. It cannot be
   re-used for a different role without new labels and retraining.
4. **Embedding model size trade-off:** `bge-small-en-v1.5` (33M params) was chosen over
   BGE-M3 due to reliability on CPU at 100k scale (Apple MPS stalls on large batches).
   Dense similarity is one of 59 features, so the quality impact is bounded.

## Caveats and Recommendations

- This system should assist, not replace, human recruiter judgment.
- Top-100 output should be reviewed by a recruiter familiar with the JD.
- If the candidate pool changes significantly (different industry, location distribution),
  recalibration is recommended.
- The reasoning strings are fact-grounded (no hallucination by construction), but they
  reflect the model's learned weights, not a full-picture assessment of the candidate.
